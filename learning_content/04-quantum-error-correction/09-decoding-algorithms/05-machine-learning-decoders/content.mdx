# Machine Learning Decoders

## Introduction

Machine learning offers a data-driven approach to quantum error correction decoding. Neural networks can learn complex syndrome-to-correction mappings, potentially adapting to real device noise better than algorithms designed for idealized error models.

## Why Machine Learning for Decoding?

```
Motivation for ML Decoders
══════════════════════════

TRADITIONAL DECODERS:
    - Designed for specific error models
    - May not match real device behavior
    - Hand-crafted algorithms

ML DECODERS:
    - Learn from data (simulated or real)
    - Can adapt to complex, correlated noise
    - End-to-end optimization


POTENTIAL ADVANTAGES:

1. ADAPTABILITY
    Real devices have:
    - Spatially varying error rates
    - Correlated errors (crosstalk)
    - Non-Markovian noise
    ML can learn these patterns!

2. SPEED
    Neural network inference is:
    - Highly parallel (GPUs/TPUs)
    - Fixed-time (no iteration)
    - Hardware-acceleratable

3. SOFT INFORMATION
    NNs naturally output probabilities
    Can provide confidence estimates


CHALLENGES:

1. TRAINING DATA
    Need many syndrome-correction pairs
    Exponential in code size?

2. GENERALIZATION
    Will it work on unseen syndromes?
    Different noise levels?

3. THRESHOLD
    Can ML match optimal threshold?
```

## Training ML Decoders

```
Training Pipeline
═════════════════

1. DATA GENERATION

    Generate training examples:
    for i in range(N_samples):
        error = sample_from_error_model(p)
        syndrome = compute_syndrome(error)
        label = compute_optimal_correction(error)
        dataset.add((syndrome, label))

    Data requirements:
    - ~10⁵ - 10⁶ samples for small codes
    - Must cover rare but important cases
    - Balance classes (most syndromes = no error)


2. LABEL GENERATION

    OPTION A: Minimum weight correction
        label = MWPM(syndrome)
        Learn to mimic MWPM

    OPTION B: Error class
        label = equivalence_class(error)
        Learn optimal decoder directly

    OPTION C: Actual error
        label = error
        May overfit to specific errors


3. LOSS FUNCTION

    Standard:
        Binary cross-entropy for each qubit
        loss = -Σ [y log(ŷ) + (1-y) log(1-ŷ)]

    QEC-aware:
        Penalize logical errors more than syndrome errors
        loss = α × logical_loss + β × syndrome_loss


4. TRAINING

    Optimizer: Adam, learning rate ~10⁻³
    Batch size: 64-256
    Epochs: until validation loss plateaus
    Regularization: dropout, weight decay


5. EVALUATION

    Metrics:
    - Per-qubit accuracy
    - Syndrome-level accuracy
    - Logical error rate (most important!)
    - Compare to MWPM baseline
```

Decoding as Reinforcement Learning

FORMULATION:

            0 otherwise

EPISODE:
    1. Sample error, compute syndrome
    2. Agent takes actions (corrections)
    3. After each action, syndrome updates
    4. Episode ends when syndrome = 0 or max steps
    5. Check for logical error, give reward


WHY RL:
    - Naturally handles sequential decisions
    - Can learn multi-step correction strategies
    - Explores vs exploits

ALGORITHMS:
    - Q-learning: learn Q(s,a) value function
    - Policy gradient: learn π(a|s) directly
    - Actor-Critic: combine both
        """)

    def advantage(self):
        """Discuss advantages of RL approach."""
        print("""
Advantages of RL Decoders
═════════════════════════

1. NO LABELED DATA NEEDED
    Reward signal provides supervision
    No need to compute optimal corrections

2. ONLINE ADAPTATION
    Can continue learning on real device
    Adapts to changing noise

3. MULTI-STEP REASONING
    Can learn complex correction sequences
    Not limited to single-shot prediction

4. EXPLORATION
    Discovers novel correction strategies
    May find better than hand-designed


CHALLENGES:

1. SAMPLE EFFICIENCY
    RL typically needs many samples
    Expensive in quantum setting

2. SPARSE REWARDS
    Most actions don't give immediate feedback
    Credit assignment problem

3. SCALABILITY
    Action space grows with code size
    State space is exponential
        """)

## Performance Comparison

```
ML Decoder Performance
══════════════════════

REPORTED RESULTS (approximate):

Decoder Type         Threshold    Inference Time    Notes
─────────────────────────────────────────────────────────────
MWPM                 ~1.0%        ~10-100 μs        Gold standard
Union-Find           ~0.9%        ~1-10 μs          Fastest
CNN                  ~0.8-0.9%    ~1-10 μs (GPU)    Needs GPU
RNN                  ~0.8%        ~10-50 μs         Sequential
RL                   ~0.7-0.8%    Variable          Sample hungry
Transformer          ~0.9%        ~10-50 μs         Recent work

KEY OBSERVATIONS:

1. THRESHOLD GAP
    ML decoders typically 0.1-0.2% below MWPM
    Gap narrows with more data and better architectures

2. SPEED
    Neural networks can be very fast on specialized hardware
    Fixed inference time (no iteration)

3. GENERALIZATION
    Mixed results on:
    - Different code distances
    - Different noise levels
    - Real vs simulated noise


HYBRID APPROACHES:

    Neural + Classical:
    - NN for initial guess, MWPM for refinement
    - NN to predict edge weights for matching
    - NN to guide Union-Find growth

    Often better than pure ML!
```

## Summary

Machine learning offers promising approaches to quantum decoding:

1. **Architectures**: Feedforward, CNN, RNN, Transformers
2. **Training**: Supervised with syndrome-correction pairs
3. **RL Approach**: Learn correction policies from rewards
4. **Performance**: ~0.8-0.9% threshold (below MWPM but competitive)
5. **Speed**: Fast inference on specialized hardware
6. **Adaptation**: Can learn device-specific noise patterns

ML decoders are an active research area with potential for future improvements.
