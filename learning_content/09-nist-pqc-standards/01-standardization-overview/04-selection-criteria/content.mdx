```python
def security_proof_requirements():
    """Understanding security proof criteria."""
    print("SECURITY PROOF REQUIREMENTS")
    print("=" * 70)

    print("""
    WHAT IS A SECURITY PROOF?
    ━━━━━━━━━━━━━━━━━━━━━━━━━━

    A security proof demonstrates that breaking the cryptosystem
    is at least as hard as solving a well-studied mathematical problem.

    ┌─────────────────────────────────────────────────────────────────┐
    │                                                                 │
    │  STRUCTURE OF A SECURITY REDUCTION:                            │
    │                                                                 │
    │  "If an adversary can break scheme S with advantage ε,         │
    │   then we can solve hard problem P with advantage ε'."         │
    │                                                                 │
    │            Break S              Solve P                        │
    │     ┌─────────────────┐   ┌─────────────────┐                  │
    │     │   Adversary A   │ → │   Algorithm B   │                  │
    │     │   (attacks S)   │   │   (solves P)    │                  │
    │     └─────────────────┘   └─────────────────┘                  │
    │                                                                 │
    │  If P is hard, then S must be secure!                          │
    │                                                                 │
    └─────────────────────────────────────────────────────────────────┘


    NIST'S PROOF EVALUATION CRITERIA:
    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    1. UNDERLYING PROBLEM HARDNESS
       ┌─────────────────────────────────────────────────────────┐
       │ • How well-studied is the hard problem?                 │
       │ • History of cryptanalytic attempts                     │
       │ • Resistance to both classical and quantum attacks      │
       │ • Concrete security estimates available                 │
       └─────────────────────────────────────────────────────────┘

    2. REDUCTION TIGHTNESS
       ┌─────────────────────────────────────────────────────────┐
       │ • How much security is "lost" in the reduction?         │
       │ • Tight reduction: ε' ≈ ε (good)                        │
       │ • Loose reduction: ε' << ε (concerning)                 │
       │ • Loose reductions require larger parameters            │
       └─────────────────────────────────────────────────────────┘

    3. PROOF MODEL
       ┌─────────────────────────────────────────────────────────┐
       │ Standard Model: Proof without idealizations (strongest) │
       │ Random Oracle Model: Hash functions are "ideal" (common)│
       │ Quantum Random Oracle Model: For PQC-specific proofs    │
       └─────────────────────────────────────────────────────────┘


    HARD PROBLEMS USED BY SELECTED ALGORITHMS:
    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    ┌────────────┬────────────────────────────────────────────────┐
    │ Algorithm  │ Hard Problem(s)                                │
    ├────────────┼────────────────────────────────────────────────┤
    │ ML-KEM     │ Module-LWE (Learning With Errors)              │
    │ ML-DSA     │ Module-LWE and Module-SIS                      │
    │ SLH-DSA    │ Hash function security only                    │
    │ FN-DSA     │ NTRU lattice problem, Short Integer Solution   │
    └────────────┴────────────────────────────────────────────────┘
    """)
```

```python
def implementation_considerations():
    """Implementation factors in selection."""
    print("\nIMPLEMENTATION CONSIDERATIONS")

    ┌─────────────────────────────────────────────────────────────────┐
    │ FACTOR                │ WHY IT MATTERS                         │
    ├───────────────────────┼────────────────────────────────────────┤
    │ Specification Clarity │ Fewer ambiguities = fewer bugs         │
    │ Code Complexity       │ Simpler code = easier to audit         │
    │ Parameter Flexibility │ Adaptability to different platforms    │
    │ Side-Channel Resist.  │ Must be securely implementable         │
    │ Error Handling        │ Failure modes must be well-defined     │
    └───────────────────────┴────────────────────────────────────────┘


    SIDE-CHANNEL RESISTANCE:
    ━━━━━━━━━━━━━━━━━━━━━━━━━

    NIST evaluated susceptibility to:

    ┌─────────────────────────────────────────────────────────────────┐
    │ ATTACK TYPE           │ DEFENSE REQUIREMENT                    │
    ├───────────────────────┼────────────────────────────────────────┤
    │ Timing Attacks        │ Constant-time implementation possible  │
    │ Power Analysis        │ Operations independent of secret data  │
    │ Cache Attacks         │ Memory access patterns don't leak      │
    │ Electromagnetic       │ Consider for high-security deploy.     │
    │ Fault Injection       │ Validation of intermediate values      │
    └───────────────────────┴────────────────────────────────────────┘


    ALGORITHM-SPECIFIC CHALLENGES:
    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    ML-KEM / ML-DSA (Lattice-based):
    ┌─────────────────────────────────────────────────────────────────┐
    │ ✓ NTT (Number Theoretic Transform) well-understood             │
    │ ✓ Polynomial arithmetic parallelizable                         │
    │ ⚠ Rejection sampling needs careful implementation              │
    │ ✓ Constant-time implementations demonstrated                   │
    └─────────────────────────────────────────────────────────────────┘

    SLH-DSA (Hash-based):
    ┌─────────────────────────────────────────────────────────────────┐
    │ ✓ Only uses hash functions - well-understood                   │
    │ ✓ Very simple to implement correctly                           │
    │ ✓ Naturally constant-time if hash function is                  │
    │ ⚠ Large state during signing (memory intensive)                │
    └─────────────────────────────────────────────────────────────────┘

    FN-DSA / FALCON (NTRU-based):
    ┌─────────────────────────────────────────────────────────────────┐
    │ ⚠ Requires floating-point or careful fixed-point               │
    │ ⚠ Gaussian sampling is tricky to implement securely            │
    │ ⚠ More complex than ML-DSA                                     │
    │ ✓ Smallest signatures among lattice schemes                    │
    └─────────────────────────────────────────────────────────────────┘
    """)
```

```python
    ├───────────────────┼────────┼────────────────────────────────────┤
    │ Flexibility       │ LOW    │ Parameter options, platform       │
    │                   │        │ adaptability                      │
    ├───────────────────┼────────┼────────────────────────────────────┤
    │ Diversity         │ LOW    │ Mathematical foundation           │

    └───────────────────┴────────┴────────────────────────────────────┘


    APPLYING THE FRAMEWORK:
    ━━━━━━━━━━━━━━━━━━━━━━━━

    When evaluating any cryptographic algorithm:

    1. SECURITY FIRST
       □ Is there a formal security proof?
       □ What problem is it reduced to?
       □ How tight is the reduction?
       □ What's the cryptanalysis history?

    2. THEN PERFORMANCE
       □ What are the operation speeds?
       □ What are the key/ciphertext/signature sizes?
       □ How does it scale with security level?

    3. THEN PRACTICALITY
       □ How complex is implementation?
       □ What are the side-channel risks?
       □ Are there reference implementations?

    4. FINALLY ECOSYSTEM
       □ What mathematical family?
       □ How does it complement other choices?
       □ What's the community support?
    """)
```

## 6. Lessons for Future Selections

```python
def future_lessons():
    """Lessons applicable to future cryptographic selections."""
    print("\nLESSONS FOR FUTURE SELECTIONS")

    1. SECURITY PROOFS ARE ESSENTIAL BUT NOT SUFFICIENT

       │ • New attacks can invalidate assumptions                    │

       │ LESSON: Proofs + extensive cryptanalysis required           │
       └─────────────────────────────────────────────────────────────┘

    2. DIVERSITY PROVIDES INSURANCE
       ┌─────────────────────────────────────────────────────────────┐
       │ • All eggs in one basket is risky                           │
       │ • Different math families hedge against unknown attacks     │
       │ • SLH-DSA selected despite performance disadvantage         │
       │                                                             │
       │ LESSON: Accept some performance cost for diversity          │
       └─────────────────────────────────────────────────────────────┘

    3. IMPLEMENTATION MATTERS AS MUCH AS THEORY
       ┌─────────────────────────────────────────────────────────────┐
       │ • Beautiful math is useless if insecure to implement        │
       │ • Side-channel resistance must be practical                 │
       │ • FALCON delayed due to implementation complexity           │
       │                                                             │
       │ LESSON: Evaluate implementation, not just algorithm         │

       │ • SIKE survived 6 years before catastrophic break           │

       │                                                             │
       │ LESSON: Patience in standardization is valuable             │
```

## Summary

NIST's selection criteria for PQC algorithms encompassed multiple dimensions:

1. **Security Proofs**: Required formal reductions to hard problems with evaluation of tightness
2. **Performance**: Measured speed and size across reference platforms
3. **Implementation**: Assessed complexity, side-channel resistance, and specification clarity
4. **Diversity**: Valued mathematical variety as insurance against future breaks

## Key Takeaways

- Security proofs are necessary but not sufficient - cryptanalysis time is crucial
- ML-KEM won due to best overall balance, not dominance in any single metric
- SLH-DSA selected for diversity despite large signatures
- Implementation complexity delayed FALCON standardization
- Future selections should apply similar multi-dimensional evaluation
