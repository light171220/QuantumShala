    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    │   are not based on structured lattices."                       │
    │                                                                 │
    │  MOTIVATION:                                                   │
    │  • ML-DSA and FN-DSA are both lattice-based                   │
    │  • SLH-DSA has large signatures (7-50 KB)                     │
    │  • Want smaller, non-lattice signatures                        │
    │                                                                 │
    │  DESIRED PROPERTIES:                                           │
    │  • Short signatures AND short public keys                      │
    │  • Fast verification                                           │
    │  • Not based on structured lattices                            │
    │  • Diverse mathematical foundations                            │
    │                                                                 │
    └─────────────────────────────────────────────────────────────────┘


    WHAT NIST IS LOOKING FOR:
    ━━━━━━━━━━━━━━━━━━━━━━━━━━

    Priority characteristics:

    1. COMPACT SIGNATURES
       • Smaller than SLH-DSA (~7-50 KB)
       • Ideally approaching ML-DSA sizes (~2.4-4.6 KB) or smaller

    2. COMPACT PUBLIC KEYS
       • Smaller than ML-DSA (~1.3-2.6 KB)
       • Small enough for certificates

    3. DIFFERENT MATHEMATICAL FOUNDATIONS
       • NOT Module-LWE/SIS (already have ML-DSA)
       • NOT NTRU lattices (already have FN-DSA)
       • Examples: MQ-based, isogeny-based, code-based, etc.


    POTENTIAL CANDIDATE FAMILIES:
    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    ┌─────────────────────────────────────────────────────────────────┐
    │ Family              │ Status                                   │
    ├─────────────────────┼──────────────────────────────────────────┤
    │ Multivariate (MQ)   │ Some schemes remain (UOV, Mayo)          │
    │ Code-based sigs     │ Active research area                     │
    │ Isogeny-based       │ Post-SIKE designs being developed        │
    │ Symmetric-based     │ Picnic-like schemes                      │
    │ Lattice (new)       │ Novel constructions welcome              │
    └─────────────────────┴──────────────────────────────────────────┘
    """)

## 4. FN-DSA (FALCON) Status

```python
def falcon_status():
    """Current status of FALCON/FN-DSA standardization."""
    print("\nFN-DSA (FALCON) STATUS")
    print("=" * 70)

    print("""
    FALCON → FN-DSA:
    ━━━━━━━━━━━━━━━━━

    FALCON was selected in July 2022 but has NOT yet been finalized.

    ┌─────────────────────────────────────────────────────────────────┐
    │                                                                 │
    │  CURRENT STATUS: Draft FIPS pending                            │
    │                                                                 │
    │  EXPECTED: Late 2024 or early 2025                             │
    │                                                                 │
    └─────────────────────────────────────────────────────────────────┘


    WHY THE DELAY?
    ━━━━━━━━━━━━━━━

    FALCON has implementation complexity issues:

    1. FLOATING-POINT ARITHMETIC
       ┌─────────────────────────────────────────────────────────────┐
       │ • Original specification uses double-precision floats       │
       │ • Floating-point behavior varies across platforms           │
       │ • Side-channel risks in FP operations                       │
       │ • Fixed-point alternatives being developed                  │
       └─────────────────────────────────────────────────────────────┘

    2. GAUSSIAN SAMPLING
       ┌─────────────────────────────────────────────────────────────┐
       │ • Requires sampling from discrete Gaussian distribution     │
       │ • Timing variations can leak secret information             │
       │ • Constant-time implementations are complex                 │
       │ • Getting this wrong breaks security                        │
       └─────────────────────────────────────────────────────────────┘

    3. FFT OPERATIONS
       ┌─────────────────────────────────────────────────────────────┐
       │ • Uses Fast Fourier Transform over complex numbers          │
       │ • Must handle numerical precision carefully                 │
       │ • Different from NTT used in ML-DSA                        │
       └─────────────────────────────────────────────────────────────┘


    FALCON'S UNIQUE VALUE:
    ━━━━━━━━━━━━━━━━━━━━━━━

    Despite complexity, FALCON offers:

    ┌─────────────────────────────────────────────────────────────────┐
    │                                                                 │
    │  SMALLEST LATTICE-BASED SIGNATURES:                            │
    │                                                                 │
    │  ┌────────────────┬─────────────┬─────────────┐                │
    │  │ Algorithm      │ Sig Size    │ Pub Key     │                │
    │  ├────────────────┼─────────────┼─────────────┤                │
    │  │ FN-DSA-512     │ ~666 bytes  │ ~897 bytes  │                │
    │  │ FN-DSA-1024    │ ~1,280 B    │ ~1,793 B    │                │
    │  │ ML-DSA-44      │ ~2,420 B    │ ~1,312 B    │                │
    │  │ ML-DSA-65      │ ~3,309 B    │ ~1,952 B    │                │
    │  └────────────────┴─────────────┴─────────────┘                │
    │                                                                 │
    │  FALCON signatures are ~2-3x smaller than ML-DSA!              │
    │                                                                 │
    └─────────────────────────────────────────────────────────────────┘

    For bandwidth-constrained applications, this matters significantly.
    """)

falcon_status()
```

## Summary

NIST's PQC standardization continues beyond the initial 2024 standards:

1. **Round 4 KEMs**: BIKE, HQC, and Classic McEliece provide code-based diversity
2. **New Signature Call**: Seeking non-lattice signatures with compact sizes
3. **FN-DSA Pending**: FALCON offers smallest lattice signatures, awaiting finalization
4. **Ongoing Evolution**: The PQC portfolio will expand over coming years

## Key Takeaways

- The current standards (ML-KEM, ML-DSA, SLH-DSA) are ready for deployment now
- Additional algorithms enhance options but don't invalidate current standards
- Crypto-agility is essential for adapting to future additions and changes
- Start migration with current standards; don't wait for a "complete" set
- Monitor standardization progress for updates relevant to your use cases
