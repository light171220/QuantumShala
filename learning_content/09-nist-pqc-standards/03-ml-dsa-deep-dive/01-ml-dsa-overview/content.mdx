# ML-DSA Overview

ML-DSA (Module-Lattice Digital Signature Algorithm) is the NIST FIPS 204 standard for post-quantum digital signatures. Formerly known as CRYSTALS-Dilithium, ML-DSA provides secure digital signatures that resist attacks from both classical and quantum computers.

## What is ML-DSA?

```
├─────────────────────────────────────────────────────────────────┤

│   Algorithm:       Module-Lattice Digital Signature Algorithm  │
│   Based On:        CRYSTALS-Dilithium                          │
│   Type:            Digital Signature                           │
│   Security:        Post-Quantum (Lattice-based)                │
```

## Digital Signatures: A Quick Review

Digital signatures provide three critical security properties:

```
┌─────────────────────────────────────────────────────────────────┐
│              DIGITAL SIGNATURE PROPERTIES                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   1. AUTHENTICATION                                             │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │  "This message came from the claimed sender"           │  │
│   │                                                         │  │
│   │  Only the holder of the private key can create        │  │
│   │  valid signatures. Anyone can verify using the        │  │
│   │  public key.                                          │  │
│   │                                                         │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   2. INTEGRITY                                                  │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │  "This message has not been modified"                  │  │
│   │                                                         │  │
│   │  Any change to the message invalidates the signature.  │  │
│   │  Even a single bit flip will cause verification       │  │
│   │  to fail.                                              │  │
│   │                                                         │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   3. NON-REPUDIATION                                            │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │  "The signer cannot deny creating the signature"       │  │
│   │                                                         │  │
│   │  Since only the private key holder can sign, they     │  │
│   │  cannot later claim they didn't create the signature. │  │
│   │                                                         │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## ML-DSA Key Components

```
┌─────────────────────────────────────────────────────────────────┐
│                   ML-DSA KEY STRUCTURE                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   PUBLIC KEY (pk):                                              │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │    pk = (ρ, t₁)                                        │  │
│   │                                                         │  │
│   │    ρ:  32-byte seed for matrix A regeneration          │  │
│   │    t₁: High-order bits of t = As₁ + s₂                │  │
│   │                                                         │  │
│   │    Used for: Signature verification                    │  │
│   │    Shared with: Everyone                               │  │
│   │                                                         │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   PRIVATE KEY (sk):                                             │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │    sk = (ρ, K, tr, s₁, s₂, t₀)                         │  │
│   │                                                         │  │
│   │    ρ:  32-byte seed for matrix A                       │  │
│   │    K:  32-byte key for signing randomness              │  │
│   │    tr: 64-byte hash of public key                      │  │
│   │    s₁: Secret polynomial vector (k elements)           │  │
│   │    s₂: Secret polynomial vector (l elements)           │  │
│   │    t₀: Low-order bits of t                             │  │
│   │                                                         │  │
│   │    Used for: Creating signatures                       │  │
│   │    Shared with: No one (keep secret!)                  │  │
│   │                                                         │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   SIGNATURE (σ):                                                │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │    σ = (c̃, z, h)                                       │  │
│   │                                                         │  │
│   │    c̃: Hashed challenge (32 bytes)                     │  │
│   │    z:  Response polynomial vector                      │  │
│   │    h:  Hint for efficient verification                │  │
│   │                                                         │  │
│   │    Used for: Proving message authenticity             │  │
│   │    Verified by: Anyone with public key                │  │
│   │                                                         │  │

│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## The Fiat-Shamir with Aborts Paradigm

ML-DSA uses a special signing approach called "Fiat-Shamir with Aborts":

```
┌─────────────────────────────────────────────────────────────────┐
│              FIAT-SHAMIR WITH ABORTS                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Standard Fiat-Shamir Transform:                               │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │   1. Commitment: Generate random y, compute w = Ay     │  │
│   │   2. Challenge:  c = H(w, message)                     │  │
│   │   3. Response:   z = y + c·s                           │  │
│   │                                                         │  │
│   │   Problem: z leaks information about s!                │  │
│   │                                                         │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   Fiat-Shamir with ABORTS (Dilithium/ML-DSA approach):         │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │   1. Commitment: Generate random y, compute w = Ay     │  │
│   │   2. Challenge:  c = H(w, message)                     │  │
│   │   3. Response:   z = y + c·s                           │  │
│   │   4. CHECK:      If z is "too large" → ABORT, restart │  │
│   │   5. Output:     If z is "small enough" → output σ    │  │
│   │                                                         │  │
│   │   Key insight: Only output signatures where z looks   │  │
│   │   independent of s. This is called "rejection sampling"│  │
│   │                                                         │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   Why Aborts Are Necessary:                                     │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │   Without aborts:                                      │  │
│   │   • z = y + c·s reveals correlation with s            │  │
│   │   • Collecting many signatures → recover s            │  │
│   │                                                         │  │
│   │   With aborts:                                         │  │
│   │   • Only output z that could have come from any s     │  │
│   │   • z is statistically independent of s               │  │
│   │   • Information-theoretic security                    │  │
│   │                                                         │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   Average Iterations:                                           │
│   ┌──────────────┬───────────────────────────────────────────┐ │
│   │   Variant    │ Expected iterations to produce signature  │ │
│   ├──────────────┼───────────────────────────────────────────┤ │
│   │ ML-DSA-44    │ ~4.25 attempts                            │ │
│   │ ML-DSA-65    │ ~5.1 attempts                             │ │
│   │ ML-DSA-87    │ ~3.85 attempts                            │ │
│   └──────────────┴───────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## Summary

ML-DSA (FIPS 204) is NIST's standardized post-quantum digital signature algorithm:

**Key Points:**
- Based on the well-studied CRYSTALS-Dilithium algorithm
- Three variants: ML-DSA-44, ML-DSA-65, ML-DSA-87
- Uses Module-LWE/Module-SIS hardness assumptions
- Employs "Fiat-Shamir with Aborts" for secure signing
- Quantum-resistant: immune to Shor's algorithm

**Trade-offs vs Classical:**
- Larger signatures (~2.5-4.5 KB vs 64-256 bytes)
- Larger public keys (~1.3-2.6 KB vs 32-256 bytes)
- Comparable or faster signing/verification speed
- Future-proof against quantum attacks

**Recommended Use:**
- ML-DSA-65 for most applications (balanced security/size)
- ML-DSA-87 for highest security requirements
- ML-DSA-44 only when size constraints are severe
