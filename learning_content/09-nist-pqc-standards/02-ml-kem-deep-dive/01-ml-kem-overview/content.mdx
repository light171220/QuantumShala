## 1. What is a Key Encapsulation Mechanism?

```python
def what_is_kem():
    """Understanding Key Encapsulation Mechanisms."""
    print("WHAT IS A KEY ENCAPSULATION MECHANISM (KEM)?")
    print("=" * 70)

    print("""
    THE PROBLEM: SHARING SECRET KEYS
    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    Alice and Bob want to communicate securely, but they've
    never met and share no secrets. How can they establish
    a shared secret key over a public channel?

    ┌─────────────────────────────────────────────────────────────────┐
    │                                                                 │
    │    Alice                    Internet                    Bob    │
    │      │                         │                         │     │
    │      │    "I want to talk      │                         │     │
    │      │     to Bob securely"    │                         │     │
    │      │──────────────────────────────────────────────────>│     │
    │      │                         │                         │     │
    │      │         Eavesdropper    │                         │     │
    │      │            Eve 👁        │                         │     │
    │      │              │          │                         │     │
    │      │         (watching)      │                         │     │
    │                                                                 │
    └─────────────────────────────────────────────────────────────────┘


    CLASSICAL SOLUTION: KEY EXCHANGE
    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    Traditional approach: Diffie-Hellman Key Exchange
    • Both parties contribute to generating shared secret
    • Based on discrete logarithm problem
    • BROKEN by quantum computers (Shor's algorithm)


    POST-QUANTUM SOLUTION: KEY ENCAPSULATION
    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    A KEM has three operations:

    ┌─────────────────────────────────────────────────────────────────┐
    │                                                                 │
    │  1. KeyGen() → (pk, sk)                                        │
    │     Generate a public key and secret key pair                  │
    │                                                                 │
    │  2. Encaps(pk) → (ct, ss)                                      │
    │     Using the public key, create:                              │
    │     • ct: ciphertext (can be sent publicly)                    │
    │     • ss: shared secret (kept private)                         │
    │                                                                 │
    │  3. Decaps(sk, ct) → ss                                        │
    │     Using the secret key and ciphertext,                       │
    │     recover the same shared secret                             │
    │                                                                 │
    └─────────────────────────────────────────────────────────────────┘


    KEM VS KEY EXCHANGE:
    ━━━━━━━━━━━━━━━━━━━━━

    ┌──────────────────┬────────────────────────────────────────────┐
    │ Key Exchange     │ Both parties contribute randomness        │
    │ (e.g., DH)       │ Interactive (back-and-forth messages)     │
    ├──────────────────┼────────────────────────────────────────────┤
    │ KEM              │ Sender generates the secret               │
    │ (e.g., ML-KEM)   │ Non-interactive (one message)             │
    │                  │ Easier to analyze security                │
    └──────────────────┴────────────────────────────────────────────┘
    """)

what_is_kem()
```

## 2. Introducing ML-KEM

```python
def introducing_ml_kem():
    """Introduction to ML-KEM and FIPS 203."""
    print("\nINTRODUCING ML-KEM (FIPS 203)")
    print("=" * 70)

    print("""
    ML-KEM: MODULE-LATTICE KEY ENCAPSULATION MECHANISM
    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    ┌─────────────────────────────────────────────────────────────────┐
    │                                                                 │
    │  FIPS 203: ML-KEM                                              │
    │  ═══════════════════                                           │
    │                                                                 │
    │  Full Name: Module-Lattice-Based Key-Encapsulation Mechanism   │
    │                                                                 │
    │  Based On:  CRYSTALS-Kyber                                     │
    │                                                                 │
    │  Published: August 13, 2024                                    │
    │                                                                 │
    │  Purpose:   Post-quantum secure key establishment              │
    │                                                                 │
    │  Replaces:  RSA key transport, ECDH, DH key exchange          │
    │                                                                 │
    └─────────────────────────────────────────────────────────────────┘


    WHY "MODULE-LATTICE"?
    ━━━━━━━━━━━━━━━━━━━━━━

    The name describes the mathematical foundation:

    • MODULE: Uses module lattices (generalization of ideal lattices)
    • LATTICE: Based on hard lattice problems

    ┌─────────────────────────────────────────────────────────────────┐
    │                                                                 │
    │  LATTICE HIERARCHY:                                            │
    │                                                                 │
    │  General Lattices                                              │
    │       │                                                        │
    │       ├── Module Lattices (ML-KEM uses this)                   │
    │       │        │                                               │
    │       │        └── Ideal Lattices (Ring-LWE)                   │
    │       │                                                        │
    │       └── Unstructured Lattices (FrodoKEM)                     │
    │                                                                 │
    │  More structure = More efficient, slightly more assumptions    │
    │                                                                 │
    └─────────────────────────────────────────────────────────────────┘


    FROM KYBER TO ML-KEM:
    ━━━━━━━━━━━━━━━━━━━━━━

    ┌─────────────────────────────────────────────────────────────────┐
    │                                                                 │
    │  CRYSTALS-Kyber (competition name)                             │
    │       │                                                        │
    │       │ Selected by NIST in July 2022                         │
    │       │                                                        │
    │       ▼                                                        │
    │  ML-KEM (FIPS 203 name)                                        │
    │                                                                 │
    │  Changes from Kyber to ML-KEM:                                 │
    │  • Minor specification clarifications                          │
    │  • Consistent naming with ML-DSA                               │
    │  • Same core algorithm                                         │
    │                                                                 │
```

```python
def ml_kem_operations():
    """The three main operations of ML-KEM."""
    print("\nML-KEM OPERATIONS")
    print("=" * 70)

    print("""
    THE THREE OPERATIONS:
    ━━━━━━━━━━━━━━━━━━━━━━

    ┌─────────────────────────────────────────────────────────────────┐
    │                                                                 │
    │  ML-KEM.KeyGen()                                               │
    │  ═══════════════                                               │
    │                                                                 │
    │  Input:  Random seed d (32 bytes)                              │
    │  Output: Encapsulation key ek (public)                         │
    │          Decapsulation key dk (secret)                         │
    │                                                                 │
    │  Purpose: Generate the key pair for one party                  │
    │                                                                 │
    │  ┌──────────────────────────────────────────────────────┐      │
    │  │  d ──────────────────────────────────────────────────│──>ek │
    │  │  │                                                    │      │
    │  │  │    (complex lattice-based key generation)         │      │
    │  │  │                                                    │      │
    │  │  └────────────────────────────────────────────────────│──>dk │
    │  └──────────────────────────────────────────────────────┘      │
    │                                                                 │
    └─────────────────────────────────────────────────────────────────┘

    ┌─────────────────────────────────────────────────────────────────┐
    │                                                                 │
    │  ML-KEM.Encaps(ek)                                             │
    │  ═════════════════                                             │
    │                                                                 │
    │  Input:  Encapsulation key ek (public key of recipient)        │
    │          Random seed m (32 bytes)                              │
    │  Output: Ciphertext c                                          │
    │          Shared secret key K (32 bytes)                        │
    │                                                                 │
    │  Purpose: Sender creates ciphertext and derives shared secret  │
    │                                                                 │
    │  ┌──────────────────────────────────────────────────────┐      │
    │  │  ek, m ──────────────────────────────────────────────│──> c │
    │  │    │                                                  │      │
    │  │    │    (encrypt random value under public key)      │      │
    │  │    │                                                  │      │
    │  │    └──────────────────────────────────────────────────│──> K │
    │  └──────────────────────────────────────────────────────┘      │
    │                                                                 │
    └─────────────────────────────────────────────────────────────────┘

    ┌─────────────────────────────────────────────────────────────────┐
    │                                                                 │
    │  ML-KEM.Decaps(dk, c)                                          │
    │  ═════════════════════                                         │
    │                                                                 │
    │  Input:  Decapsulation key dk (secret key)                     │
    │          Ciphertext c (received from sender)                   │
    │  Output: Shared secret key K (32 bytes)                        │
    │                                                                 │
    │  Purpose: Recipient recovers the same shared secret            │
    │                                                                 │
    │  ┌──────────────────────────────────────────────────────┐      │
    │  │  dk, c ──────────────────────────────────────────────│──> K │
    │  │     │                                                 │      │
    │  │     │    (decrypt and verify to recover secret)      │      │
    │  │     │                                                 │      │
    │  └──────────────────────────────────────────────────────┘      │
    │                                                                 │
    │  CRITICAL: K from Encaps must equal K from Decaps             │
    │                                                                 │
    └─────────────────────────────────────────────────────────────────┘
    """)
```

```python
def ml_kem_high_level():
    """High-level explanation of ML-KEM operation."""
    print("\nHOW ML-KEM WORKS (HIGH LEVEL)")
    print("=" * 70)

    print("""
    THE PROTOCOL FLOW:
    ━━━━━━━━━━━━━━━━━━━

    ┌─────────────────────────────────────────────────────────────────┐
    │                                                                 │
    │  Bob (Recipient)                      Alice (Sender)           │
    │  ═══════════════                      ══════════════           │
    │                                                                 │
    │  1. KeyGen()                                                   │
    │     (ek, dk) ← ML-KEM.KeyGen()                                │
    │                                                                 │
    │  2. Share public key                                           │
    │     ────────────────────────────────────────>                  │
    │     ek (encapsulation key, public)                             │
    │                                                                 │
    │                                       3. Encapsulate           │
    │                                          (c, K) ← Encaps(ek)  │
    │                                                                 │
    │  4. Receive ciphertext                                         │
    │     <────────────────────────────────────────                  │
    │     c (ciphertext)                                             │
    │                                                                 │
    │  5. Decapsulate                                                │
    │     K ← Decaps(dk, c)                                         │
    │                                                                 │
    │  ═══════════════════════════════════════════════════════════   │
    │  RESULT: Both parties now share secret K                       │
    │          Eve (eavesdropper) cannot compute K                   │
    │  ═══════════════════════════════════════════════════════════   │
    │                                                                 │
    └─────────────────────────────────────────────────────────────────┘


    THE UNDERLYING MATHEMATICS (SIMPLIFIED):
    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    ┌─────────────────────────────────────────────────────────────────┐
    │                                                                 │
    │  Key Idea: Learning With Errors (LWE) Problem                  │
    │                                                                 │
    │  PUBLIC:   A (random matrix), b = A·s + e (noisy product)     │
    │  SECRET:   s (secret vector), e (small error vector)          │
    │                                                                 │
    │  HARD PROBLEM:                                                 │
    │  Given A and b, find s                                         │
    │  (The small error e makes this hard!)                          │
    │                                                                 │
    │  ┌─────────────────────────────────────────────────────────┐   │
    │  │                                                         │   │
    │  │  Without error:  b = A·s  → Easy (linear algebra)      │   │
    │  │                                                         │   │
    │  │  With error:     b = A·s + e  → HARD (no known attack) │   │
    │  │                                                         │   │
    │  └─────────────────────────────────────────────────────────┘   │
    │                                                                 │
    │  ML-KEM uses Module-LWE: A more efficient structured version   │
```

## Summary

ML-KEM (FIPS 203) is the primary post-quantum key encapsulation mechanism:

1. **What it is**: A KEM based on the Module-LWE problem, evolved from CRYSTALS-Kyber
2. **Operations**: KeyGen, Encaps, Decaps - enabling secure shared secret establishment
3. **Security**: Three parameter sets for Categories 1, 3, and 5
4. **Use Cases**: TLS, VPN, SSH, secure messaging, and hybrid encryption

## Key Takeaways

- ML-KEM replaces RSA/DH/ECDH for key establishment
- The shared secret is always 32 bytes regardless of parameter set
- ML-KEM-768 (Category 3) is recommended for most applications
- Hybrid deployment (ECDH + ML-KEM) recommended during transition
- The underlying Module-LWE problem is believed hard for quantum computers
