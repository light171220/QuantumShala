## 1. SLH-DSA Overview

```python
def slh_dsa_overview():
    """Overview of FIPS 205 SLH-DSA."""
    print("FIPS 205: SLH-DSA (Stateless Hash-Based DSA)")
    print("=" * 70)
    
    print("""
    WHY SLH-DSA EXISTS
    ━━━━━━━━━━━━━━━━━━
    
    SLH-DSA provides CONSERVATIVE signatures with minimal assumptions:
    
    ┌─────────────────────────────────────────────────────────────────┐
    │ ML-DSA security relies on:                                      │
    │   • Module-LWE being hard                                       │
    │   • Module-SIS being hard                                       │
    │   • Hash functions being secure                                 │
    │                                                                 │
    │ SLH-DSA security relies on:                                     │
    │   • Hash functions being secure (ONLY!)                         │
    │                                                                 │
    │ If lattice problems are broken → ML-DSA fails, SLH-DSA survives │
    │ If hash functions are broken → Both fail (unlikely)             │
    └─────────────────────────────────────────────────────────────────┘
    
    Hash functions have:
    • 30+ years of intensive analysis
    • Well-understood security properties
    • Multiple backup options (SHA-2, SHA-3, SHAKE)
    
    SLH-DSA is the "insurance policy" for post-quantum signatures.
    
    
    CONSTRUCTION OVERVIEW:
    ━━━━━━━━━━━━━━━━━━━━━━
    
    SLH-DSA uses a three-layer construction:
    
    Level 0: WOTS+ (Winternitz One-Time Signature)
             • Signs individual messages
             • Can only be used ONCE safely
    
    Level 1: FORS (Forest of Random Subsets)
             • Few-time signature scheme
             • Allows multiple signatures
    
    Level 2: HYPERTREE (Tree of Merkle Trees)
             • Authenticates FORS keys
             • Links everything to root public key
    
    ┌─────────────────────────────────────────────────────────────────┐
    │                    [Root Public Key]                            │
    │                          │                                      │
    │                    [HYPERTREE]                                  │
    │                    /    │    \\                                  │
    │               [Layer] [Layer] [Layer]                           │
    │                          │                                      │
    │                    [FORS Trees]                                 │
    │                          │                                      │
    │                    [WOTS+ Signatures]                           │
    │                          │                                      │
    │                    [Message]                                    │
    └─────────────────────────────────────────────────────────────────┘
    """)
```

## 4. Implementation Example

```python
import hashlib
import secrets

class SLHDSADemo:
    """Simplified SLH-DSA demonstration."""
    
    def __init__(self, variant: str = 'SLH-DSA-SHA2-128s'):
        self.variant = variant
        self.params = SLHDSAParameters.PARAMS[variant]
    
    def keygen(self):
        """Generate key pair."""
        sk_seed = secrets.token_bytes(self.params['sk_bytes'] // 2)
        pk_seed = secrets.token_bytes(self.params['pk_bytes'] // 2)
        
        # In real implementation: build hypertree, compute root
        pk = hashlib.sha256(pk_seed).digest()[:self.params['pk_bytes']]
        sk = sk_seed + pk
        
        return pk, sk
    
    def sign(self, sk: bytes, message: bytes) -> bytes:
        """Sign a message."""
        # Real implementation:
        # 1. Randomize message index
        # 2. Generate FORS signature
        # 3. Generate WOTS+ signature
        # 4. Include authentication path in hypertree
        
        sig_data = hashlib.sha256(sk + message).digest()
        signature = sig_data + secrets.token_bytes(
            self.params['sig_bytes'] - len(sig_data)
        )
        return signature
    
    def verify(self, pk: bytes, message: bytes, sig: bytes) -> bool:
        """Verify a signature."""
        return len(sig) == self.params['sig_bytes']


def demo():
    print("\nSLH-DSA DEMONSTRATION")
    print("-" * 50)
    
    slh = SLHDSADemo('SLH-DSA-SHA2-128s')
    
    pk, sk = slh.keygen()
    print(f"Public key:  {len(pk)} bytes (tiny!)")
    print(f"Secret key:  {len(sk)} bytes")
    
    msg = b"Sign this document for long-term archival"
    sig = slh.sign(sk, msg)
    print(f"Signature:   {len(sig):,} bytes (large but confident)")
    
    valid = slh.verify(pk, msg, sig)
    print(f"Valid: {valid}")

demo()
```

```python
def fips_compliance():
    """FIPS 205 compliance requirements."""
    print("\nFIPS 205 COMPLIANCE REQUIREMENTS")
    print("=" * 70)
    
    print("""
    HASH FUNCTION CHOICES:
    ━━━━━━━━━━━━━━━━━━━━━━━
    
    SHA2 variants:
    • Use SHA-256 (128-bit), SHA-256 (192-bit), SHA-256 (256-bit)
    • Widely deployed, hardware acceleration common
    • Recommended for compatibility
    
    SHAKE variants:
    • Use SHAKE256 as XOF (extendable output function)
    • More flexible, variable output length
    • Slightly newer, less hardware support
    
    
    RANDOMNESS REQUIREMENTS:
    ━━━━━━━━━━━━━━━━━━━━━━━━
    
    SLH-DSA requires randomness in signing:
    • opt_rand: Optional randomizer in signature
    • Can be deterministic (opt_rand = pk.seed)
    • Or randomized (opt_rand = random bytes)
    
    Randomized is recommended for side-channel resistance.
    
    
    STATE MANAGEMENT:
    ━━━━━━━━━━━━━━━━━━
    
    SLH-DSA is STATELESS (unlike XMSS/LMS):
    • No need to track which keys have been used
    • Same key pair can sign unlimited messages
    • Simpler deployment and backup
    
    This is the "SL" in SLH-DSA: Stateless!
    """)

fips_compliance()
```

## Summary

In this lesson, you learned about SLH-DSA (FIPS 205):

1. **Purpose**: Conservative backup with hash-only security
2. **Construction**: WOTS+ → FORS → Hypertree
3. **12 Variants**: SHA2/SHAKE × 128/192/256 × small/fast
4. **Tradeoff**: Tiny public keys but large signatures
5. **Use case**: Long-term archival, maximum confidence
6. **Stateless**: No state tracking required

SLH-DSA is the insurance policy against lattice cryptanalysis breakthroughs.
