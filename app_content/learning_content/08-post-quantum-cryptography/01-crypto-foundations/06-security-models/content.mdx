## 1. Why Formal Security Models Matter

Cryptography is unique among security disciplines: we can actually *prove* that systems are secure, given certain assumptions. But what does "secure" mean exactly?

<Definition term="Security Model">
A formal mathematical framework that precisely defines what it means for a cryptographic system to be secure, including the capabilities of the adversary and the security goals.
</Definition>

### The Problem with Informal Security

```
"This cipher is unbreakable!"
    ↓
Questions that need answers:
• Unbreakable by whom? (computational power)
• Under what conditions? (known plaintext? chosen plaintext?)
• What exactly can't they do? (recover key? decrypt? forge?)
• Based on what assumptions? (factoring is hard? random oracle?)
```

<Warning>
Throughout history, "secure" systems without formal analysis have been broken. The Enigma machine, WEP wireless encryption, and countless others fell because security wasn't precisely defined and proven.
</Warning>

## 2. Adversary Models

Security is always relative to an adversary's capabilities. We formalize this:

### Adversary Capabilities

<Definition term="Adversary Model">
A specification of what resources and capabilities an attacker has, including computational power, access to the system, and knowledge about the scheme.
</Definition>

| Model | Description | Example |
|-------|-------------|---------|
| **Ciphertext-only** | Adversary sees only ciphertexts | Intercepting encrypted emails |
| **Known-plaintext** | Adversary knows some plaintext-ciphertext pairs | Standard encryption attacks |
| **Chosen-plaintext (CPA)** | Adversary can encrypt chosen messages | Access to encryption oracle |
| **Chosen-ciphertext (CCA)** | Adversary can also decrypt chosen ciphertexts | Access to decryption oracle |

```python
class AdversaryModel:
    """Conceptual representation of adversary capabilities"""
    
    def __init__(self, model_type: str):
        self.model_type = model_type
        
    def ciphertext_only(self):
        """Weakest adversary: only sees ciphertexts"""
        return {
            'can_see': ['ciphertexts'],
            'can_do': [],
            'goal': 'learn anything about plaintexts'
        }
    
    def known_plaintext(self):
        """Adversary knows some plaintext-ciphertext pairs"""
        return {
            'can_see': ['ciphertexts', 'some_plaintexts'],
            'can_do': [],
            'goal': 'decrypt new ciphertexts or recover key'
        }
    
    def chosen_plaintext(self):
        """Adversary can encrypt arbitrary messages"""
        return {
            'can_see': ['ciphertexts', 'some_plaintexts'],
            'can_do': ['encrypt_chosen_messages'],
            'goal': 'distinguish encryptions or recover key'
        }
    
    def chosen_ciphertext(self):
        """Strongest adversary: can also request decryptions"""
        return {
            'can_see': ['ciphertexts', 'some_plaintexts'],
            'can_do': ['encrypt_chosen_messages', 'decrypt_chosen_ciphertexts'],
            'goal': 'decrypt challenge ciphertext'
        }
```

<Tip>
Always design for the strongest reasonable adversary model. If your scheme is secure against CCA attacks, it's automatically secure against weaker attacks.
</Tip>

## 3. Security Definitions for Encryption

### IND-CPA: Indistinguishability under Chosen-Plaintext Attack

<Definition term="IND-CPA">
A scheme is IND-CPA secure if no efficient adversary, given access to an encryption oracle, can distinguish between encryptions of two chosen messages with probability significantly better than 1/2.
</Definition>

<Formula>
$$\text{Advantage}_{\text{IND-CPA}}(A) = \left| \Pr[A \text{ wins}] - \frac{1}{2} \right| \leq \text{negligible}$$
</Formula>

```
IND-CPA Game:
┌─────────────────────────────────────────────────────────┐
│ 1. Challenger generates key pair (pk, sk)              │
│ 2. Adversary gets pk and access to Encrypt(pk, ·)      │
│ 3. Adversary chooses two messages m₀, m₁               │
│ 4. Challenger flips coin b, sends c* = Encrypt(pk, mᵦ) │
│ 5. Adversary outputs guess b'                          │
│ 6. Adversary wins if b' = b                            │
└─────────────────────────────────────────────────────────┘

Security: Adversary can't do better than random guessing (50%)
```

```python
def ind_cpa_game_simulation():
    """
    Simulate IND-CPA security game (conceptual)
    """
    import random
    
    # Challenger setup
    key = generate_key()
    
    # Adversary chooses two messages
    m0 = b"Message zero"
    m1 = b"Message one!"  # Same length!
    
    # Challenger's challenge
    b = random.randint(0, 1)
    challenge_ciphertext = encrypt(key, [m0, m1][b])
    
    # Adversary tries to guess which message was encrypted
    # For a secure scheme, adversary can only guess randomly
    adversary_guess = random.randint(0, 1)  # Random guess
    
    return adversary_guess == b  # ~50% success rate for secure schemes
```

### IND-CCA: Indistinguishability under Chosen-Ciphertext Attack

<Definition term="IND-CCA">
A scheme is IND-CCA secure if no efficient adversary can distinguish encryptions even with access to a decryption oracle (except for the challenge ciphertext itself).
</Definition>

```
IND-CCA Game:
┌─────────────────────────────────────────────────────────┐
│ 1. Challenger generates key pair (pk, sk)              │
│ 2. Adversary gets pk, Encrypt(pk, ·), and Decrypt(sk, ·)│
│ 3. Adversary chooses two messages m₀, m₁               │
│ 4. Challenger sends c* = Encrypt(pk, mᵦ)               │
│ 5. Adversary can still query Decrypt(sk, c) for c ≠ c* │
│ 6. Adversary outputs guess b'                          │
└─────────────────────────────────────────────────────────┘

Stronger than IND-CPA: adversary has decryption oracle!
```

<Warning>
**Textbook RSA is NOT IND-CPA secure!** It's deterministic: encrypting the same message twice gives the same ciphertext. This is why we use padding schemes like OAEP.
</Warning>

### Comparison of Security Levels

| Property | Meaning | Achieved By |
|----------|---------|-------------|
| IND-CPA | Can't distinguish encryptions | Randomized encryption |
| IND-CCA1 | IND-CPA + decryption oracle before challenge | Some padding schemes |
| IND-CCA2 | IND-CPA + decryption oracle after challenge | OAEP, GCM |

## 4. Security Definitions for Signatures

### EUF-CMA: Existential Unforgeability under Chosen-Message Attack

<Definition term="EUF-CMA">
A signature scheme is EUF-CMA secure if no efficient adversary, even with access to a signing oracle, can produce a valid signature on any message not previously signed.
</Definition>

```
EUF-CMA Game:
┌─────────────────────────────────────────────────────────┐
│ 1. Challenger generates key pair (pk, sk)              │
│ 2. Adversary gets pk and access to Sign(sk, ·)         │
│ 3. Adversary queries Sign on messages m₁, m₂, ...      │
│ 4. Adversary outputs (m*, σ*)                          │
│ 5. Adversary wins if Verify(pk, m*, σ*) = true         │
│    AND m* ∉ {m₁, m₂, ...}                              │
└─────────────────────────────────────────────────────────┘

Security: Can't forge signature on ANY new message
```

```python
def euf_cma_game_simulation():
    """
    Simulate EUF-CMA security game (conceptual)
    """
    # Challenger setup
    private_key, public_key = generate_keypair()
    
    # Adversary's queries (tracked by challenger)
    signed_messages = []
    
    def signing_oracle(message):
        """Adversary can request signatures on any message"""
        signature = sign(private_key, message)
        signed_messages.append(message)
        return signature
    
    # Adversary uses oracle
    sig1 = signing_oracle(b"Message 1")
    sig2 = signing_oracle(b"Message 2")
    
    # Adversary attempts forgery
    forged_message = b"Forged message"
    forged_signature = adversary_forge(public_key, signed_messages)
    
    # Check if forgery succeeds
    forgery_valid = verify(public_key, forged_message, forged_signature)
    message_is_new = forged_message not in signed_messages
    
    adversary_wins = forgery_valid and message_is_new
    return adversary_wins  # Should be False for secure scheme
```

### Signature Security Levels

| Level | Description |
|-------|-------------|
| **EUF-CMA** | Can't forge signature on new message |
| **SUF-CMA** | Can't forge new signature even on signed message |
| **UUF-CMA** | Can't forge on random message (weaker) |

## 5. Computational vs Information-Theoretic Security

### Computational Security

<Definition term="Computational Security">
Security that holds against adversaries with bounded computational resources. Breaking the scheme is possible in principle but requires infeasible amounts of computation.
</Definition>

```python
# Computational security example
# AES-256 is computationally secure

key_space = 2**256  # Possible keys
operations_per_second = 10**12  # 1 trillion
seconds_per_year = 365.25 * 24 * 60 * 60

years_to_brute_force = key_space / operations_per_second / seconds_per_year
print(f"Years to brute-force AES-256: {years_to_brute_force:.2e}")
# Output: 3.67e+57 years (computationally infeasible)
```

### Information-Theoretic Security

<Definition term="Information-Theoretic Security">
Security that holds against adversaries with unlimited computational power. Even with infinite time, the adversary cannot break the scheme.
</Definition>

<Example>
**One-Time Pad: Perfect Secrecy**

```python
import os

def one_time_pad_encrypt(plaintext: bytes) -> tuple:
    """
    One-time pad provides perfect (information-theoretic) secrecy.
    Ciphertext reveals ZERO information about plaintext.
    """
    key = os.urandom(len(plaintext))  # Key as long as message
    ciphertext = bytes(p ^ k for p, k in zip(plaintext, key))
    return key, ciphertext

def demonstrate_perfect_secrecy():
    """
    Any plaintext is equally likely given the ciphertext.
    """
    ciphertext = bytes([0x48, 0x65, 0x6c, 0x6c, 0x6f])
    
    # This ciphertext could decrypt to ANY 5-character message
    # depending on the key. With no knowledge of the key,
    # "Hello" and "World" are equally likely.
    
    # Key that decrypts to "Hello"
    key_hello = bytes(c ^ ord(p) for c, p in zip(ciphertext, "Hello"))
    
    # Key that decrypts to "World"  
    key_world = bytes(c ^ ord(p) for c, p in zip(ciphertext, "World"))
    
    print(f"Same ciphertext, different keys:")
    print(f"Key {key_hello.hex()} → 'Hello'")
    print(f"Key {key_world.hex()} → 'World'")

demonstrate_perfect_secrecy()
```
</Example>

### Comparison

| Aspect | Computational | Information-Theoretic |
|--------|---------------|----------------------|
| **Against** | Bounded adversary | Unbounded adversary |
| **Examples** | AES, RSA | One-time pad, secret sharing |
| **Assumptions** | Problem hardness | None (unconditional) |
| **Practicality** | Practical key sizes | Often impractical |
| **Quantum impact** | May be broken | Remains secure |

## 6. Security Reductions

<Definition term="Security Reduction">
A proof technique showing that if an adversary can break scheme A, they can also solve problem B. If B is believed hard, A must be secure.
</Definition>

<Formula>
$$\text{If } A \text{ is broken} \Rightarrow B \text{ is solved}$$
$$\text{Contrapositive: } B \text{ is hard} \Rightarrow A \text{ is secure}$$
</Formula>

```
Security Reduction Structure:

┌─────────────────────────────────────────────────────────┐
│                    Assumed Hard Problem                 │
│                    (e.g., Factoring)                   │
└─────────────────────────────────────────────────────────┘
                          ↑
                     Reduction
                          ↑
┌─────────────────────────────────────────────────────────┐
│                    Cryptographic Scheme                 │
│                    (e.g., RSA)                         │
└─────────────────────────────────────────────────────────┘

"Breaking RSA is at least as hard as factoring"
```

<Example>
**RSA Security Reduction (Simplified)**

```python
def rsa_reduction_concept():
    """
    Conceptual: If someone can break RSA, they can factor.
    
    This is a simplification; the actual reduction is more nuanced.
    """
    
    # Suppose adversary A can decrypt any RSA ciphertext
    def rsa_adversary(N, e, ciphertext):
        """Hypothetical adversary that breaks RSA"""
        return "decrypted_message"  # Magic!
    
    # We can use A to factor N
    def factor_using_rsa_adversary(N, e):
        """
        Use RSA adversary to factor N.
        
        Key insight: If we can recover d from (N, e),
        we can factor N.
        """
        # Choose random message
        m = random_integer(2, N-1)
        
        # Encrypt it
        c = pow(m, e, N)
        
        # Use adversary to decrypt
        m_recovered = rsa_adversary(N, e, c)
        
        # If adversary works, we can deduce d and factor N
        # (actual math omitted for clarity)
        return "factors_of_N"
    
    # Conclusion: If RSA is broken, factoring is solved
    # Contrapositive: If factoring is hard, RSA is secure
```
</Example>

### Quality of Reductions

| Reduction Type | Meaning |
|---------------|---------|
| **Tight** | Breaking scheme ≈ solving hard problem |
| **Loose** | Breaking scheme << solving hard problem |
| **Non-tight** | Large security loss in reduction |

<Tip>
Post-quantum schemes often have tighter reductions than classical schemes, which is a significant advantage for security confidence.
</Tip>

## 7. Security Parameters and Levels

### Security Levels (NIST)

| Level | Classical Security | Quantum Security | Reference |
|-------|-------------------|------------------|-----------|
| 1 | 128 bits | 64 bits | AES-128 |
| 2 | 192 bits | 96 bits | SHA-384 |
| 3 | 192 bits | 128 bits | AES-192 |
| 4 | 256 bits | 128 bits | SHA-512 |
| 5 | 256 bits | 256 bits | AES-256 |

## 8. Post-Quantum Security Considerations

### New Hard Problems

Post-quantum cryptography relies on different hard problems:

| Problem | Used By | Security Basis |
|---------|---------|----------------|
| **LWE** | Kyber, Dilithium | Lattice problems |
| **Module-LWE** | Kyber, Dilithium | Structured lattices |
| **NTRU** | NTRU, Falcon | Lattice problems |
| **Syndrome Decoding** | McEliece, BIKE | Coding theory |
| **Hash functions** | SPHINCS+ | Hash security |

### Security Model Considerations for PQC


## Summary

In this lesson, you learned about security models:

1. **Formal models** precisely define security and enable proofs
2. **Adversary models** (CPA, CCA) specify attacker capabilities
3. **IND-CPA/IND-CCA** define encryption security (indistinguishability)
4. **EUF-CMA** defines signature security (unforgeability)
5. **Computational security** relies on problem hardness assumptions
6. **Information-theoretic security** is unconditional but often impractical
7. **Security reductions** prove scheme security based on hard problems
8. **Post-quantum schemes** rely on new hard problems with strong reductions
