## 3. Evaluation Criteria

```python
def evaluation_criteria():
    """NIST's evaluation criteria for PQC algorithms."""
    print("\nNIST EVALUATION CRITERIA")
    print("=" * 70)
    
    print("""
    PRIMARY CRITERIA:
    ━━━━━━━━━━━━━━━━━━
    
    1. SECURITY (Most Important)
    ┌─────────────────────────────────────────────────────────────────┐
    │ • Resistance to classical attacks                              │
    │ • Resistance to quantum attacks (Shor's, Grover's)             │
    │ • Security proof quality and tightness                         │
    │ • Conservative security margins                                │
    │ • Side-channel resistance potential                            │
    └─────────────────────────────────────────────────────────────────┘
    
    2. PERFORMANCE
    ┌─────────────────────────────────────────────────────────────────┐
    │ • Computational speed (keygen, encrypt, sign)                  │
    │ • Memory requirements                                          │
    │ • Bandwidth (key and ciphertext/signature sizes)               │
    │ • Suitability for constrained devices                          │
    └─────────────────────────────────────────────────────────────────┘
    
    3. ALGORITHM CHARACTERISTICS
    ┌─────────────────────────────────────────────────────────────────┐
    │ • Flexibility of parameters                                    │
    │ • Simplicity of design                                         │
    │ • Quality of documentation                                     │
    │ • Clarity of implementation requirements                       │
    └─────────────────────────────────────────────────────────────────┘
    
    
    SECURITY CATEGORIES:
    ━━━━━━━━━━━━━━━━━━━━━
    
    NIST defined 5 security levels based on attack difficulty:
    
    ┌─────────────────────────────────────────────────────────────────┐
    │ Category │ Classical Equivalent │ Quantum Equivalent           │
    ├──────────┼──────────────────────┼──────────────────────────────┤
    │ 1        │ AES-128 key search   │ Grover on AES-128            │
    │ 2        │ SHA-256 collision    │ Quantum collision on SHA-256 │
    │ 3        │ AES-192 key search   │ Grover on AES-192            │
    │ 4        │ SHA-384 collision    │ Quantum collision on SHA-384 │
    │ 5        │ AES-256 key search   │ Grover on AES-256            │

    WHAT NIST WAS LOOKING FOR:
    ━━━━━━━━━━━━━━━━━━━━━━━━━━━

    • MATURITY: Well-analyzed constructions
    • PRACTICALITY: Deployable in real systems
    • CONSERVATIVE: Security margins for unknowns
    """)
```

## 4. The Final Standards

```python
def final_standards():
    """The three final NIST PQC standards."""
    print("\nFINAL NIST PQC STANDARDS (August 2024)")
    print("=" * 70)
    
    print("""
    ┌─────────────────────────────────────────────────────────────────┐
    │                     FIPS 203: ML-KEM                           │
    │            Module-Lattice Key Encapsulation Mechanism          │
    ├─────────────────────────────────────────────────────────────────┤
    │ Based on:     CRYSTALS-Kyber                                   │
    │ Purpose:      Key establishment (key exchange)                 │
    │ Foundation:   Module-LWE problem                               │
    │ Variants:     ML-KEM-512, ML-KEM-768, ML-KEM-1024             │
    │ Security:     Categories 1, 3, 5                               │
    │ Primary use:  TLS, VPN, secure messaging                       │
    └─────────────────────────────────────────────────────────────────┘
    
    ┌─────────────────────────────────────────────────────────────────┐
    │                     FIPS 204: ML-DSA                           │
    │           Module-Lattice Digital Signature Algorithm           │
    ├─────────────────────────────────────────────────────────────────┤
    │ Based on:     CRYSTALS-Dilithium                               │
    │ Purpose:      Digital signatures                               │
    │ Foundation:   Module-LWE and Module-SIS problems               │
    │ Variants:     ML-DSA-44, ML-DSA-65, ML-DSA-87                  │
    │ Security:     Categories 2, 3, 5                               │
    │ Primary use:  Code signing, certificates, documents            │
    └─────────────────────────────────────────────────────────────────┘
    
    ┌─────────────────────────────────────────────────────────────────┐
    │                     FIPS 205: SLH-DSA                          │
    │           Stateless Hash-Based Digital Signature Algorithm     │
    ├─────────────────────────────────────────────────────────────────┤
    │ Based on:     SPHINCS+                                         │
    │ Purpose:      Digital signatures (conservative backup)         │
    │ Foundation:   Hash function security only                      │
    │ Variants:     12 parameter sets (SHA2/SHAKE × size × speed)   │
    │ Security:     Categories 1, 3, 5                               │
    │ Primary use:  Root CAs, long-term signing, high assurance     │
    └─────────────────────────────────────────────────────────────────┘
    
    
    WHY THREE STANDARDS?
    ━━━━━━━━━━━━━━━━━━━━━
    
    ML-KEM:  For key exchange (replacing ECDH, RSA key transport)
    ML-DSA:  Primary signature (replacing RSA, ECDSA signatures)
    SLH-DSA: Conservative backup (different math, maximum confidence)
    
    
    WHAT'S STILL COMING:
    ━━━━━━━━━━━━━━━━━━━━━
    
    • FALCON: Compact lattice signatures (draft expected 2024-2025)
    • Additional signatures from new call (evaluation ongoing)
    • Classic McEliece: Code-based KEM (under consideration)
    """)

final_standards()
```

## 5. Lessons from the Process

```python
def lessons_learned():
    """Key lessons from the NIST PQC process."""
    print("\nLESSONS FROM NIST PQC PROCESS")
    print("=" * 70)
    
    print("""
    KEY TAKEAWAYS:
    ━━━━━━━━━━━━━━━
    
    1. OPEN COMPETITION WORKS
       • 82 submissions from worldwide experts
       • Public analysis improved security
       • Transparent evaluation built trust
    
    2. CRYPTANALYSIS IS ESSENTIAL
       • SIKE broken after years of analysis
       • Rainbow eliminated in Round 3
       • Continuous improvement of survivors
    
    3. DIVERSITY IS VALUABLE
       • Lattice-based (ML-KEM, ML-DSA)
       • Hash-based (SLH-DSA)
       • Code-based still being evaluated
    
    4. PRACTICAL PERFORMANCE MATTERS
       • Huge keys are problematic
       • Speed affects real deployments
       • Balance security with usability
    
    5. STANDARDIZATION TAKES TIME
       • 8 years from call to final standards
       • Rushing risks security flaws
       • Patience produces better results
    
    
    THE SIKE LESSON:
    ┌─────────────────────────────────────────────────────────────────┐
    │ SIKE was a NIST Round 3 alternate candidate.                   │
    │ In August 2022, it was COMPLETELY BROKEN by a classical        │
    │ polynomial-time attack—after years of analysis.                │
    │                                                                 │
    │ Lesson: Even extensively studied schemes can fall.             │
    │ This is why:                                                   │
    │   • Multiple algorithm families are needed                     │
    │   • Hybrid approaches provide insurance                        │
    │   • Cryptographic agility is essential                         │
    └─────────────────────────────────────────────────────────────────┘
    
    
    WHAT THIS MEANS FOR YOU:
    ━━━━━━━━━━━━━━━━━━━━━━━━━
    
    ✓ Standards are ready—start migration planning
    ✓ Use ML-KEM + ML-DSA for most applications
    ✓ Consider SLH-DSA for highest assurance needs
    ✓ Implement hybrid (classical + PQC) during transition
    ✓ Design for crypto agility (algorithms may change)
    """)

lessons_learned()
```

## Summary

The NIST PQC standardization process:

1. **8 Years**: From 2016 call to 2024 final standards
2. **82 → 3**: Rigorous evaluation narrowed the field
3. **Three Standards**: ML-KEM, ML-DSA, SLH-DSA (FIPS 203/204/205)
4. **Lessons**: Open competition, cryptanalysis essential, diversity valuable
5. **Action**: Standards are ready—begin migration now
