## 1. Why Hybrid Cryptography?

```python
def why_hybrid():
    """The case for hybrid cryptography."""
    print("WHY HYBRID CRYPTOGRAPHY?")
    print("=" * 70)
    
    print("""
    THE DILEMMA:
    ━━━━━━━━━━━━
    
    Classical Cryptography (RSA, ECC):
    • Decades of cryptanalysis
    • Well-understood security margins
    • Proven track record
    • WILL BE BROKEN by quantum computers
    
    Post-Quantum Cryptography (ML-KEM, ML-DSA):
    • Relatively new (years of analysis, not decades)
    • Less cryptanalysis history
    • Larger keys and signatures
    • MIGHT have undiscovered weaknesses
    
    
    THE SOLUTION: DEFENSE IN DEPTH
    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    
    Combine BOTH classical and post-quantum:
    
    ┌─────────────────────────────────────────────────────────────────┐
    │                                                                 │
    │  If classical breaks:    PQC still protects you               │
    │  If PQC breaks:          Classical still protects you         │
    │  If quantum arrives:     PQC protects you                     │
    │                                                                 │
    │  HYBRID SECURITY = MAX(classical, post-quantum)               │
    │                                                                 │
    └─────────────────────────────────────────────────────────────────┘
    
    This is called "defense in depth" - if one layer fails,
    the other still provides protection.
    
    
    REAL-WORLD MOTIVATION:
    ━━━━━━━━━━━━━━━━━━━━━━━
    
    Remember SIKE/SIDH?
    • NIST Round 3 finalist
    • Considered secure until August 2022
    • Broken by polynomial-time classical attack!
    
    If you had deployed SIKE-only, you'd be scrambling.
    If you had deployed SIKE+ECDH hybrid, ECDH still protects you.
    """)

why_hybrid()
```

## 2. Hybrid Design Principles

```python
def hybrid_principles():
    """Core principles of hybrid cryptographic design."""
    print("\nHYBRID DESIGN PRINCIPLES")
    print("=" * 70)
    
    print("""
    PRINCIPLE 1: COMBINE, DON'T REPLACE
    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    
    During transition, ADD post-quantum to classical.
    Don't remove classical until PQC is fully trusted.
    
    WRONG: Replace ECDH with ML-KEM
    RIGHT: Use ECDH AND ML-KEM together
    
    
    PRINCIPLE 2: BOTH MUST BE SECURE
    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    
    The hybrid should be secure if EITHER component is secure.
    Breaking one component should not help break the other.
    
    Security(Hybrid) ≥ max(Security(Classical), Security(PQC))
    
    
    PRINCIPLE 3: PROPER KEY COMBINATION
    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    
    WRONG: K = K_classical || K_pqc (simple concatenation)
    
    RIGHT: K = KDF(K_classical || K_pqc || context)
    
    Use a proper key derivation function (KDF) to combine:
    • Ensures uniform output
    • Provides domain separation
    • Binds to context
    
    
    PRINCIPLE 4: INDEPENDENT COMPONENTS
    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    
    Classical and PQC should use:
    • Different mathematical assumptions
    • Independent random values
    • Separate key material
    
    No shared secrets between components!
    
    
    PRINCIPLE 5: FAIL SECURELY
    ━━━━━━━━━━━━━━━━━━━━━━━━━━━
    
    If one component fails:
    • Log the failure (for monitoring)
    • Continue with the other component
    • Or abort safely (depending on policy)
    """)

hybrid_principles()
```

## 3. Hybrid Key Exchange

```python
import hashlib
import secrets

class HybridKEM:
    """
    X25519 + ML-KEM-768 Hybrid Key Encapsulation.
    Simplified for educational purposes.
    """
    
    def __init__(self):
        self.label = b"X25519-MLKEM768-Hybrid-v1"
    
    def combine_secrets(self, k_classical: bytes, k_pqc: bytes,
                       context: bytes = b"") -> bytes:
        """
        Combine classical and PQC shared secrets using HKDF.
        
        Per NIST guidance:
        - Use approved KDF (HKDF)
        - Include both secrets
        - Add domain separation
        """
        # Concatenate inputs
        combined = k_classical + k_pqc + context + self.label
        
        # Apply KDF (simplified - real implementation uses HKDF)
        return hashlib.sha256(combined).digest()
    
    def demonstrate(self):
        """Demonstrate hybrid key exchange."""
        print("\nHYBRID KEY EXCHANGE DEMONSTRATION")
        print("-" * 50)
        
        # Simulate two parties
        # In reality, these would be X25519 and ML-KEM outputs
        
        # Classical component (X25519)
        k_classical = secrets.token_bytes(32)
        print(f"Classical shared secret: {k_classical.hex()[:32]}...")
        
        # PQC component (ML-KEM-768)
        k_pqc = secrets.token_bytes(32)
        print(f"PQC shared secret:       {k_pqc.hex()[:32]}...")
        
        # Context (e.g., hash of ciphertexts for binding)
        context = secrets.token_bytes(32)
        
        # Combine using KDF
        k_hybrid = self.combine_secrets(k_classical, k_pqc, context)
        print(f"Hybrid shared secret:    {k_hybrid.hex()[:32]}...")
        
        print("""
        
    KEY PROPERTIES:
    • If attacker breaks classical only: still needs PQC key
    • If attacker breaks PQC only: still needs classical key
    • Quantum attacker: faces ML-KEM (designed to be quantum-safe)
    • Classical attacker: faces both (even harder)
        """)

kem = HybridKEM()
kem.demonstrate()
```

## Summary

Hybrid cryptography provides essential defense in depth:

1. **Rationale**: Protect against both classical and quantum threats
2. **Principles**: Combine properly, use KDF, maintain independence
3. **Overhead**: Minimal vs PQC-only (+32-64 bytes)
4. **SIKE Lesson**: Even analyzed algorithms can break unexpectedly
5. **Timeline**: Use hybrid now, evaluate PQC-only later
6. **Key insight**: Better to have two defenses than one

When in doubt, use hybrid—the cost is minimal, the protection is maximal.
