## 3. Implementation Details

```python
def implementation_details():
    """Technical implementation details for TLS PQC."""
    print("\nIMPLEMENTATION DETAILS")
    print("=" * 70)
    
    print("""
    KEY SHARE STRUCTURE:
    ━━━━━━━━━━━━━━━━━━━━━
    
    KeyShareEntry {
        NamedGroup group;           // e.g., X25519MLKEM768
        uint16 key_exchange_length;
        opaque key_exchange<1..2^16-1>;
    }
    
    For X25519MLKEM768:
    key_exchange = X25519_public (32 B) || MLKEM768_public (1184 B)
    
    
    SERVER RESPONSE:
    ━━━━━━━━━━━━━━━━━
    
    Server returns:
    key_exchange = X25519_public (32 B) || MLKEM768_ciphertext (1088 B)
    
    Note: Server sends ciphertext, not public key for ML-KEM!
    
    
    SHARED SECRET DERIVATION:
    ━━━━━━━━━━━━━━━━━━━━━━━━━━
    """)
    
    print('''
    ```python
    def derive_hybrid_shared_secret(
        my_x25519_sk: bytes,
        their_x25519_pk: bytes,
        my_mlkem_sk: bytes,
        mlkem_ciphertext: bytes
    ) -> bytes:
        """Derive hybrid shared secret for TLS 1.3."""
        
        # Classical component
        ss_classical = x25519_shared_secret(my_x25519_sk, their_x25519_pk)
        
        # PQC component  
        ss_pqc = ml_kem_decapsulate(my_mlkem_sk, mlkem_ciphertext)
        
        # Combine (per IETF draft)
        combined = ss_classical + ss_pqc
        
        # This becomes the (EC)DHE input to TLS key schedule
        return combined
    
    
    def tls13_key_schedule(combined_secret: bytes):
        """TLS 1.3 key schedule with hybrid input."""
        
        # Early secret (from PSK or zeros)
        early_secret = hkdf_extract(salt=b'', ikm=b'\\x00' * 32)
        
        # Handshake secret
        derived = derive_secret(early_secret, "derived", b"")
        handshake_secret = hkdf_extract(salt=derived, ikm=combined_secret)
        
        # ... continue with standard TLS 1.3 key schedule
        return handshake_secret
    ```
    ''')

implementation_details()
```

## 4. Certificate Chain Considerations

```python
def certificate_considerations():
    """Certificate chain considerations for PQC TLS."""
    print("\nCERTIFICATE CHAIN CONSIDERATIONS")
    print("=" * 70)
    
    print("""
    THE CERTIFICATE SIZE PROBLEM:
    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
    
    Classical Certificate Chain (typical):
    ┌─────────────────────────────────────────────────────────────────┐
    │ • End-entity cert:  ~1.5 KB (RSA-2048 signature)               │
    │ • Intermediate cert: ~1.5 KB                                   │
    │ • Root cert: (not sent, but ~1.5 KB)                           │
    │ • Total: ~3 KB                                                 │
    └─────────────────────────────────────────────────────────────────┘
    
    PQC Certificate Chain (ML-DSA-65):
    ┌─────────────────────────────────────────────────────────────────┐
    │ • End-entity cert:  ~5.3 KB (ML-DSA-65 sig + public key)       │
    │ • Intermediate cert: ~5.3 KB                                   │
    │ • Root cert: (not sent)                                        │
    │ • Total: ~10-12 KB                                             │
    └─────────────────────────────────────────────────────────────────┘
    
    Increase: ~3-4x larger certificate chains!
    
    
    IMPACT ON TLS HANDSHAKE:
    ━━━━━━━━━━━━━━━━━━━━━━━━━
    
    ┌─────────────────────────────────────────────────────────────────┐
    │ Component              │ Classical    │ PQC/Hybrid             │
    ├────────────────────────┼──────────────┼────────────────────────┤
    │ Key exchange           │ ~100 B       │ ~2.3 KB                │
    │ Certificate chain      │ ~3 KB        │ ~10-12 KB              │
    │ Handshake signature    │ ~256 B       │ ~3.3 KB                │
    ├────────────────────────┼──────────────┼────────────────────────┤
    │ Total handshake        │ ~4 KB        │ ~18 KB                 │
    └────────────────────────────────────────────────────────────────┘
    
    
    MITIGATION STRATEGIES:
    ━━━━━━━━━━━━━━━━━━━━━━━
    
    1. CERTIFICATE COMPRESSION (RFC 8879)
       • Compress certificate chain
       • 50-70% reduction possible
       • Requires client support
    
    2. DELEGATED CREDENTIALS (RFC 9345)
       • Short-lived PQC credential
       • Signed by classical certificate
       • Reduces chain impact
    
    3. CACHED CERTIFICATES
       • Client caches intermediate certs
       • Server only sends end-entity
       • Requires prior connection
    
    4. SUPPRESS UNNECESSARY CERTS
       • Don't send root cert
       • Optimize chain length
    """)

certificate_considerations()
```

## Summary

PQC in TLS 1.3 protects against "harvest now, decrypt later":

1. **Hybrid Key Exchange**: X25519 + ML-KEM-768 combined
2. **Size Impact**: ~2.3 KB additional for key exchange
3. **Certificate Challenge**: 3-4x larger chains, use compression
4. **Browser Support**: Chrome enabled, others following
5. **Deployment**: Phased rollout with fallback capability

Start with hybrid—it protects against both classical and quantum attackers.
