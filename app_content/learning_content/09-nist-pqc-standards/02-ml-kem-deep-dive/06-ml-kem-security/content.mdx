# ML-KEM Security Analysis

Understanding ML-KEM's security is essential for making informed deployment decisions. This lesson provides a comprehensive analysis of ML-KEM's security foundations, known attack vectors, side-channel considerations, and long-term security outlook.

## The Module-LWE Problem

ML-KEM's primary security assumption is the hardness of Module-LWE:

```
┌─────────────────────────────────────────────────────────────────┐
│                  MODULE-LWE PROBLEM                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Parameters:                                                   │
│   • n = 256 (polynomial degree)                                │
│   • k = 2, 3, or 4 (module rank)                               │
│   • q = 3329 (modulus)                                         │
│   • χ = centered binomial distribution (error distribution)   │
│                                                                 │
│   Problem Statement:                                            │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │  Given: A ∈ R_q^{k×k}, b = A·s + e ∈ R_q^k             │  │
│   │  Where: R_q = Z_q[X]/(X^n + 1)                          │  │
│   │         s ← χ^k (secret)                                │  │
│   │         e ← χ^k (error)                                 │  │
│   │                                                         │  │
│   │  Distinguish from: (A, uniform random b ∈ R_q^k)       │  │
│   │                                                         │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   Why It's Hard:                                                │
│   • The error e masks the relationship between A, s, and b    │
│   • Without knowing e, solving for s is as hard as SVP       │
│   • The ring structure R_q provides efficiency but limited   │
│     exploitable structure                                      │
│                                                                 │
│   Module-LWE vs Variants:                                       │
│   ┌──────────────┬─────────────┬───────────────────────────┐  │
│   │   Variant    │  Structure  │  Security Assessment      │  │
│   ├──────────────┼─────────────┼───────────────────────────┤  │
│   │   LWE        │  None       │  Most conservative        │  │
│   │   Ring-LWE   │  Maximum    │  Fastest, some concerns   │  │
│   │   Module-LWE │  Moderate   │  Best balance ✓           │  │
│   └──────────────┴─────────────┴───────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## Security Reductions

ML-KEM's security is proven through a chain of reductions:

```
┌─────────────────────────────────────────────────────────────────┐
│                  SECURITY REDUCTION CHAIN                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Theorem (Informal): If Module-LWE is hard, ML-KEM is         │
│   IND-CCA2 secure.                                              │
│                                                                 │
│   Reduction Steps:                                              │
│                                                                 │
│   Step 1: K-PKE IND-CPA ← Module-LWE                           │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │  An adversary breaking K-PKE can solve Module-LWE:     │  │
│   │                                                         │  │
│   │  • Encapsulation key ek = (A, t = As + e) looks like   │  │
│   │    Module-LWE samples                                   │  │
│   │  • Ciphertext (u, v) also relies on Module-LWE         │  │
│   │  • Breaking CPA means distinguishing these from random │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   Step 2: ML-KEM IND-CCA2 ← K-PKE IND-CPA + ROM               │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │  The Fujisaki-Okamoto transform provides:              │  │
│   │                                                         │  │
│   │  • Hash function G modeled as random oracle            │  │
│   │  • Re-encryption check ensures ciphertext validity     │  │
│   │  • Implicit rejection prevents oracle attacks          │  │
│   │                                                         │  │
│   │  Result: CPA security "lifts" to CCA2 security        │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   Security Loss (Tightness):                                    │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │  ε_CCA2 ≤ q_H · ε_CPA + q_D · δ                        │  │
│   │                                                         │  │
│   │  Where:                                                 │  │
│   │  • q_H = number of hash queries                        │  │
│   │  • q_D = number of decapsulation queries              │  │
│   │  • δ = decryption failure probability (negligible)    │  │
│   │                                                         │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## Known Attack Vectors

### Lattice Attacks

The primary attack class against ML-KEM involves lattice reduction:

```
┌─────────────────────────────────────────────────────────────────┐
│                  LATTICE ATTACKS                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   1. BKZ Algorithm (Classical)                                  │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │  Block Korkine-Zolotarev lattice reduction              │  │
│   │                                                         │  │
│   │  • Best known classical algorithm for SVP              │  │
│   │  • Running time: 2^{O(n)} for dimension n              │  │
│   │  • Practical limit: dimension ~400-500                 │  │
│   │                                                         │  │
│   │  ML-KEM lattice dimension: k × 256 × 2 = 1024-2048    │  │
│   │  → Classical BKZ is infeasible                        │  │
│   │                                                         │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   2. Lattice Sieving (Quantum)                                 │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │  Quantum algorithms for lattice problems:               │  │
│   │                                                         │  │
│   │  • Grover's algorithm for sieving enumeration          │  │
│   │  • Best known: 2^{0.265n} quantum operations          │  │
│   │  • Provides roughly √ speedup over classical          │  │
│   │                                                         │  │
│   │  ML-KEM security estimates include quantum attacks    │  │
│   │                                                         │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   Core-SVP Security Estimates:                                  │
│   ┌──────────────┬────────────┬────────────┬─────────────┐    │
│   │   Variant    │ Classical  │  Quantum   │ NIST Level  │    │
│   ├──────────────┼────────────┼────────────┼─────────────┤    │
│   │ ML-KEM-512   │  ~2^150    │  ~2^118    │ Category 1  │    │
│   │ ML-KEM-768   │  ~2^200    │  ~2^161    │ Category 3  │    │
│   │ ML-KEM-1024  │  ~2^250    │  ~2^200    │ Category 5  │    │
│   └──────────────┴────────────┴────────────┴─────────────┘    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

## Side-Channel Attacks

Implementation security is as critical as mathematical security:

```
┌─────────────────────────────────────────────────────────────────┐
│                  SIDE-CHANNEL ATTACKS                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │

│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │

│   │  • Ciphertext comparison in decapsulation             │  │
│   │  • Polynomial coefficient processing                   │  │
│   │  • Conditional branches based on secret data          │  │
│   │                                                         │  │
│   │  Mitigations (required by FIPS 203):                   │  │
│   │  • Constant-time comparison functions                 │  │

│   │  • Branch-free conditional selection                   │  │

│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   2. POWER ANALYSIS                                             │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │  Attack Vectors:                                        │  │
│   │  • Simple Power Analysis (SPA) on NTT operations      │  │

│   │  • Electromagnetic emanations                          │  │
│   │                                                         │  │

│   │  • Randomized NTT computation                          │  │

│                                                                 │
│   3. FAULT ATTACKS                                              │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │

│   │  • Modify ciphertext processing                        │  │
│   │                                                         │  │
│   │  Mitigations:                                           │  │
│   │  • Redundant computation                               │  │
│   │  • Integrity checks on intermediate values            │  │

│   │                                                         │  │

│   │                                                         │  │
│   │  Mitigations:                                           │  │

## Decryption Failure Analysis

ML-KEM has a small but non-zero decryption failure probability:

```
┌─────────────────────────────────────────────────────────────────┐
│              DECRYPTION FAILURE PROBABILITY                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Failure Mechanism:                                            │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │  During decryption, we compute:                        │  │
│   │                                                         │  │
│   │    μ' = v - s^T·u                                      │  │
│   │       = μ + (e₂ + s^T·e₁ - e^T·r) + compression_noise  │  │
│   │                                                         │  │
│   │  If |noise| > q/4, decoding produces wrong bit        │  │
│   │                                                         │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   Noise Sources:                                                │
│   ┌────────────────────────┬────────────────────────────────┐  │
│   │ Source                 │ Contribution                   │  │
│   ├────────────────────────┼────────────────────────────────┤  │
│   │ e₂ (encryption error) │ ~η₂ (CBD parameter)            │  │
│   │ s^T·e₁               │ ~k·n·η₁·η₂                       │  │
│   │ e^T·r                 │ ~k·n·η₁·η₂                       │  │
│   │ Compression (u)       │ ~q/2^{du+1}                     │  │
│   │ Compression (v)       │ ~q/2^{dv+1}                     │  │
│   └────────────────────────┴────────────────────────────────┘  │
│                                                                 │
│   Failure Probabilities:                                        │
│   ┌──────────────┬───────────────────────────────────────────┐ │
│   │   Variant    │ Failure Probability                       │ │
│   ├──────────────┼───────────────────────────────────────────┤ │
│   │ ML-KEM-512   │ < 2^{-139} per ciphertext                │ │
│   │ ML-KEM-768   │ < 2^{-164} per ciphertext                │ │
│   │ ML-KEM-1024  │ < 2^{-174} per ciphertext                │ │
│   └──────────────┴───────────────────────────────────────────┘ │
│                                                                 │
│   Security Implications:                                        │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │  • Failure probability << 2^{-128} is acceptable       │  │
│   │  • Attacker cannot induce failures (implicit reject)  │  │
│   │  • Natural failures extremely rare in practice         │  │
│   │  • FO transform handles failures securely              │  │
│   │                                                         │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## Summary

ML-KEM's security rests on solid foundations:

**Mathematical Security:**
- Based on Module-LWE, a well-studied computational problem
- Security reduces to the hardness of lattice problems (SVP)
- Conservative parameters provide security margins above targets
- No known quantum algorithms threaten lattice problems

**Implementation Security:**
- Constant-time operations prevent timing attacks
- Implicit rejection secures against oracle attacks
- FO transform provides IND-CCA2 from IND-CPA
- FIPS 203 specifies security requirements

**Security Recommendations:**
- ML-KEM-768 for most applications (good balance)
- ML-KEM-1024 for high-sensitivity, long-term data
- ML-KEM-512 only with classical algorithm hybrid
- Always use validated, constant-time implementations

**Ongoing Considerations:**
- Monitor lattice cryptanalysis research
- Plan for algorithm agility
- Consider hybrid deployments during transition
- Regular security assessments of implementations
