# SLH-DSA Overview

SLH-DSA (Stateless Hash-Based Digital Signature Algorithm) is the NIST FIPS 205 standard for post-quantum digital signatures. Based on SPHINCS+, it provides the most conservative security assumptions among NIST's post-quantum signature standards, relying only on the security of hash functions.

## What is SLH-DSA?

```
┌─────────────────────────────────────────────────────────────────┐
│                      SLH-DSA AT A GLANCE                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Standard:        FIPS 205                                    │
│   Algorithm:       Stateless Hash-Based Digital Signature      │
│   Based On:        SPHINCS+                                    │
│   Type:            Digital Signature                           │
│   Security:        Hash Function Security Only                 │
│   Assumption:      Second-preimage resistance of hash          │
│                                                                 │
│   Key Operations:                                               │
│   ┌────────────┬─────────────────────────────────────────────┐ │
│   │  KeyGen    │ Generate public/private key pair            │ │
│   │  Sign      │ Create signature using Merkle trees         │ │
│   │  Verify    │ Verify signature against public key         │ │
│   └────────────┴─────────────────────────────────────────────┘ │
│                                                                 │
│   Parameter Sets (12 variants):                                │
│   ┌────────────────┬───────────────┬─────────────────────────┐ │
│   │ Security Level │ Fast (f)      │ Small (s)               │ │
│   ├────────────────┼───────────────┼─────────────────────────┤ │
│   │ Category 1     │ SLH-DSA-128f  │ SLH-DSA-128s           │ │
│   │ Category 3     │ SLH-DSA-192f  │ SLH-DSA-192s           │ │
│   │ Category 5     │ SLH-DSA-256f  │ SLH-DSA-256s           │ │
│   └────────────────┴───────────────┴─────────────────────────┘ │
│   (Also SHAKE and SHA2 variants for each)                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## Why Hash-Based Signatures?

```
┌─────────────────────────────────────────────────────────────────┐
│              WHY HASH-BASED SIGNATURES?                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   The Conservative Choice:                                      │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │   ML-DSA/ML-KEM security relies on:                    │  │
│   │   • Hardness of lattice problems (MLWE, MSIS)          │  │
│   │   • ~15 years of cryptanalysis                         │  │
│   │   • Algebraic structure that could be exploited        │  │
│   │                                                         │  │
│   │   SLH-DSA security relies on:                          │  │
│   │   • Hash function security (SHA-256, SHAKE)            │  │
│   │   • ~30+ years of cryptanalysis                        │  │
│   │   • No algebraic structure to exploit                  │  │
│   │                                                         │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   Security Confidence Hierarchy:                                │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │   HIGHEST   Hash functions (SHA-256, SHA-3)            │  │
│   │      │      - Decades of analysis                      │  │
│   │      │      - Minimal mathematical structure           │  │
│   │      │      - Used everywhere, heavily scrutinized    │  │
│   │      │                                                  │  │
│   │      ▼                                                  │  │
│   │   HIGH      Lattice problems (LWE, SIS)                │  │
│   │      │      - 15+ years of analysis                    │  │
│   │      │      - Some algebraic structure                 │  │
│   │      │      - No known quantum speedups               │  │
│   │      │                                                  │  │
│   │      ▼                                                  │  │
│   │   MEDIUM    Code-based, isogeny-based                  │  │
│   │             - Various levels of analysis               │  │
│   │             - Some recent breaks (isogenies)          │  │
│   │                                                         │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## Stateless vs Stateful Signatures

```
┌─────────────────────────────────────────────────────────────────┐
│              STATELESS VS STATEFUL                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Stateful Hash-Based Signatures (XMSS, LMS):                  │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │   • Must track which one-time keys have been used     │  │
│   │   • Reusing a key BREAKS security completely          │  │
│   │   • Smaller signatures (~2-3 KB)                      │  │
│   │   • Requires secure, persistent state storage         │  │
│   │   • Limited number of signatures per key              │  │
│   │                                                         │  │
│   │   Risk: State loss or reuse → total key compromise   │  │
│   │                                                         │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   Stateless (SLH-DSA/SPHINCS+):                                │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │   • No state to track or synchronize                  │  │
│   │   • Can sign unlimited messages with same key         │  │
│   │   • Larger signatures (~7-50 KB)                      │  │
│   │   • Simpler to implement securely                     │  │
│   │   • Drop-in replacement for classical signatures      │  │
│   │                                                         │  │
│   │   Trade-off: Larger signatures for simpler security  │  │
│   │                                                         │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
│   Why Stateless Matters:                                        │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                                                         │  │
│   │   Scenario: Certificate Authority signs certificates   │  │
│   │                                                         │  │
│   │   Stateful:                                             │  │
│   │   • Must maintain counter across restarts             │  │
│   │   • Database corruption = key compromise              │  │
│   │   • Load balancing requires state synchronization     │  │
│   │   • Backup/restore is dangerous                       │  │
│   │                                                         │  │
│   │   Stateless (SLH-DSA):                                 │  │
│   │   • Just need the private key                         │  │
│   │   • Can sign on any machine with key access          │  │
│   │   • No synchronization needed                         │  │
│   │   • Standard backup procedures work                   │  │
│   │                                                         │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## Summary

SLH-DSA (FIPS 205) key characteristics:

**Design Philosophy:**
- Most conservative security assumptions (hash functions only)
- Stateless operation - no state synchronization needed
- Trade signature size for security confidence

**Comparison with ML-DSA:**
- Larger signatures (~7-50 KB vs ~3 KB)
- Much smaller public keys (32-64 bytes vs ~2 KB)
- Slower signing and verification
- Higher security confidence

**Best Use Cases:**
- Root CA certificates
- Firmware signing
- Long-term document signatures
- Defense-in-depth with ML-DSA

**Parameter Selection:**
- "f" variants: Faster signing, larger signatures
- "s" variants: Smaller signatures, slower signing
- SHA-2 and SHAKE instantiations available
