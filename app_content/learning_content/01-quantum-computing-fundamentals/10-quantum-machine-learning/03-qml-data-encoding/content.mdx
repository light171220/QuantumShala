# Data Encoding for QML

## Introduction

Before quantum circuits can process classical data, that data must be encoded into quantum states. This encoding step is crucial and can significantly impact algorithm performance. Different encoding strategies offer different tradeoffs between qubit count, circuit depth, and expressiveness.

## The Encoding Challenge

```
The Data Encoding Bottleneck
════════════════════════════

PROBLEM:
    Classical data lives in R^n
    Quantum states live in Hilbert space
    We need to map: x ∈ R^n → |ψ(x)⟩ ∈ C^(2^q)

WHY IT MATTERS:
    ┌─────────────────────────────────────────────────────────┐
    │                                                         │
    │   Classical     Encoding      Quantum      Quantum     │
    │   Data      →   Circuit   →   State   →   Processing  │
    │                                                         │
    │   O(N)           O(?)         Ready        O(polylog)  │
    │   data           depth                     processing  │
    │                                                         │
    └─────────────────────────────────────────────────────────┘

    If encoding takes O(N) time/depth,
    we lose any quantum speedup!

TRADEOFFS:
    - Qubits: How many qubits needed?
    - Depth: How deep is the encoding circuit?
    - Expressiveness: How much information encoded?
    - Trainability: Does it help or hurt learning?
```

## Basis Encoding

```python
def basis_encoding_demo():
    """
    Demonstrate basis encoding of classical data.
    """
    print("""
Basis Encoding
══════════════

CONCEPT:
    Encode integer i as computational basis state |i⟩
    Binary representation directly → qubit states

EXAMPLE:
    Data: [0, 3, 5, 7]
    Qubits: 3 (to represent up to 7)

    0 → |000⟩
    3 → |011⟩
    5 → |101⟩
    7 → |111⟩

SUPERPOSITION OF DATA:
    Encode dataset as superposition:
    |ψ⟩ = (1/2)(|000⟩ + |011⟩ + |101⟩ + |111⟩)


CIRCUIT:
    For integer x with bits b₀b₁b₂...:

    |0⟩ ───[X if b₀]───
    |0⟩ ───[X if b₁]───
    |0⟩ ───[X if b₂]───


PROPERTIES:
    ┌─────────────────────────────────────────────┐
    │  Metric       │  Value                     │
    ├─────────────────────────────────────────────┤
    │  Qubits       │  log₂(max value)           │
    │  Depth        │  O(1) per data point       │
    │  Gate count   │  O(n) X gates              │
    │  Expressiveness│  Low (discrete values)    │
    └─────────────────────────────────────────────┘

USE CASE:
    - Encoding indices
    - Discrete/categorical data
    - Simple but limited
    """)

basis_encoding_demo()
```

## Amplitude Encoding

```
Amplitude Encoding
══════════════════

CONCEPT:
    Encode vector x ∈ R^N as amplitudes of quantum state
    |ψ⟩ = Σᵢ xᵢ|i⟩ / ||x||

EXAMPLE:
    Vector x = [0.5, 0.5, 0.5, 0.5]
    Normalized: Already unit norm

    |ψ⟩ = 0.5|00⟩ + 0.5|01⟩ + 0.5|10⟩ + 0.5|11⟩
        = |++⟩

    Uses only 2 qubits to encode 4 numbers!


EFFICIENCY:
    N classical numbers → log₂(N) qubits
    Exponential compression!

    ┌─────────────────────────────────────────────┐
    │  Data size N  │  Qubits needed             │
    ├─────────────────────────────────────────────┤
    │  4            │  2                          │
    │  256          │  8                          │
    │  1,000,000    │  20                         │
    └─────────────────────────────────────────────┘


THE CATCH:
    Preparing arbitrary amplitudes is expensive!
    General state preparation: O(N) gates
    We lose the exponential advantage

    ┌─────────────────────────────────────────────┐
    │                                             │
    │  Classical       Quantum       Processing  │
    │  O(N) data  →   O(N) prep →   O(log N)    │
    │                                             │
    │  Total: O(N), not faster than classical!   │
    └─────────────────────────────────────────────┘


WHEN USEFUL:
    - Data has special structure (allows fast prep)
    - QRAM hypothetically available
    - Data prepared during quantum process
    - Repeated use of same data (amortized)


CIRCUIT (general preparation):
    Uses O(2^n) controlled rotations
    Very deep for large vectors
```

## Angle/Rotation Encoding

```python
def angle_encoding():
    """
    Explain angle-based data encoding.
    """
    print("""
Angle (Rotation) Encoding
═════════════════════════

CONCEPT:
    Encode each feature as rotation angle on separate qubit
    x_i → rotation of i-th qubit by angle ∝ x_i

CIRCUIT:
    |0⟩ ─[Ry(x₁)]─
    |0⟩ ─[Ry(x₂)]─
    |0⟩ ─[Ry(x₃)]─
    ...

    Each qubit: |0⟩ → cos(x/2)|0⟩ + sin(x/2)|1⟩


VARIATIONS:

1. SINGLE ROTATION:
    |0⟩ ─[Ry(xᵢ)]─
    1 feature per qubit

2. MULTI-ROTATION (IQP-style):
    |0⟩ ─[H]─[Rz(xᵢ)]─●─[Rz(xᵢxⱼ)]─
                      │
    |0⟩ ─[H]─[Rz(xⱼ)]─X─────────────

    Includes feature interactions

3. DATA RE-UPLOADING:
    |0⟩ ─[U(x)]─[V(θ)]─[U(x)]─[V(θ)]─...

    Re-encode data multiple times with trainable layers


PROPERTIES:
    ┌─────────────────────────────────────────────┐
    │  Metric        │  Value                    │
    ├─────────────────────────────────────────────┤
    │  Qubits        │  n (one per feature)      │
    │  Depth         │  O(1) to O(d) for depth d │
    │  Gate count    │  O(n) to O(n²)            │
    │  Expressiveness│  Medium to High           │
    └─────────────────────────────────────────────┘

ADVANTAGES:
    ✓ Simple circuits
    ✓ Depth independent of data size
    ✓ Naturally handles continuous data
    ✓ Works well with variational methods

MOST COMMON in near-term QML!
    """)

angle_encoding()
```

## Feature Maps

```
Quantum Feature Maps
════════════════════

CONCEPT:
    Embedding classical data into quantum Hilbert space
    Feature map: x → |φ(x)⟩

PURPOSE:
    Create "quantum feature space" where:
    - Linear separability may emerge
    - Quantum correlations encode relationships
    - Kernel methods become powerful


COMMON FEATURE MAPS:

1. ZZ FEATURE MAP:
    |0⟩ ─[H]─[Rz(x₁)]─●─[Rz(x₁x₂)]─[H]─[Rz(x₁)]─●─
                      │                          │
    |0⟩ ─[H]─[Rz(x₂)]─X──────────────[H]─[Rz(x₂)]─X─

    Repeated layers with ZZ interactions


2. PAULI FEATURE MAP:
    |0⟩ ─[H]─[P(x)]─[entangle]─[H]─[P(x)]─...

    P(x) = product of Pauli rotations


3. IQP (Instantaneous Quantum Polynomial):
    Diagonal gates sandwiched by Hadamards
    |0⟩ ─[H]─[Rz(poly(x))]─[H]─


CHOOSING A FEATURE MAP:

    Consider:
    - Data dimension vs qubits
    - Desired feature interactions
    - Circuit depth limitations
    - Kernel properties needed

    ┌─────────────────────────────────────────────┐
    │  Feature Map    │  Best For                │
    ├─────────────────────────────────────────────┤
    │  Simple angle   │  Low-dimensional data    │
    │  ZZ feature map │  Feature interactions    │
    │  IQP            │  Theoretical guarantees  │
    │  Custom         │  Domain knowledge        │
    └─────────────────────────────────────────────┘
```

## Summary

Data encoding strategies for QML:

1. **Basis encoding**: Simple, good for discrete/index data
2. **Amplitude encoding**: Exponential compression, but expensive to prepare
3. **Angle encoding**: Practical for near-term, one qubit per feature
4. **Feature maps**: Rich quantum feature spaces for kernel methods

The choice of encoding significantly impacts both quantum resources and learning performance.
